Deployment Setup Plan

This project should be moved from the developer-owned accounts to accounts owned by the final project owner.
The goal is that the owner controls the source code, hosting, database, realtime service, domain, and future storage.

Required owner accounts

The owner should create and provide access to these accounts:

1. GitHub
   - Create a private repository for this project.
   - The project source code will be pushed to that repository.
   - The developer can be added as a collaborator only if maintenance access is needed.

2. Vercel
   - Create a Vercel account or team under the owner's ownership.
   - Connect the owner's GitHub repository to Vercel.
   - Create a Vercel project from that repository.
   - Add the required environment variables in the Vercel project settings.

3. Supabase
   - Create a Supabase project for the production database.
   - This will replace the current local PostgreSQL database for production.
   - Supabase Storage is used for public listing images.

4. Ably
   - Create an Ably account and app for realtime calendar updates.
   - Add the Ably API key to Vercel environment variables.

5. Domain and DNS
   - The owner should buy or control the production domain.
   - Recommended DNS provider: Cloudflare.
   - The domain will be connected to the Vercel project.

6. Domain email
   - If the owner needs email addresses like info@domain.com, use a domain email provider.
   - Recommended options: Zoho Mail for lower cost, or Google Workspace for Gmail-style business email.

How this project will be linked to the owner accounts

1. GitHub link
   - Create a new private repository under the owner's GitHub account or organization.
   - Change this local project's Git remote to the owner's repository.
   - Push the project source code to the owner's repository.
   - Confirm that future pushes go only to the owner's repository.

2. Vercel link
   - In the owner's Vercel account, import the owner's GitHub repository.
   - Configure the build settings for the Next.js project.
   - Add all required production environment variables.
   - Deploy the project from the owner's Vercel account.

3. Supabase database link
   - Create the Supabase project.
   - Copy the Supabase PostgreSQL connection string.
   - Add the Supabase database URL to Vercel environment variables.
   - Run the database schema on Supabase.
   - Seed the starting data if needed.
   - Test login, calendar updates, approvals, owners, venues, and space types against Supabase.

4. Ably realtime link
   - Create an Ably app under the owner's Ably account.
   - Copy the Ably API key.
   - Add the Ably API key to Vercel environment variables.
   - Test realtime calendar sync between two browser sessions.

5. Domain link
   - Add the owner's domain to Vercel.
   - Update DNS records in the domain provider or Cloudflare.
   - Confirm the production site opens correctly over HTTPS.

Database migration plan

1. Keep the local database for development only.
2. Create a new Supabase production database.
3. Run the current database schema on Supabase.
4. Provision any required initial production users with reviewed one-time SQL.
   The demo seed/reset scripts are local-only and must never run in production.
5. Update Vercel environment variables to use Supabase.
6. Deploy the project.
7. Test all important flows:
   - Login
   - Owner access
   - Superadmin access
   - Calendar save
   - Owner change request
   - Approval/rejection
   - Pending request deletion
   - Owner activation/deactivation
   - Venue activation/deactivation
   - Space type restrictions
   - Realtime calendar sync

Image storage note

Public listing images use Supabase Storage in production.
Create a public bucket named listing-images in the same Supabase project used by DATABASE_URL.
Set the bucket file-size limit to 4194304 bytes and allow only image/jpeg, image/png, and image/webp.
Uploads and deletions go through the authenticated server route, so do not add public insert, update, or delete policies.
Add SUPABASE_URL and the server-only SUPABASE_SECRET_KEY to Vercel Production environment variables.
SUPABASE_SERVICE_ROLE_KEY remains supported only as a legacy fallback.
Add a long random CRON_SECRET so Vercel can authenticate the daily abandoned-image cleanup job.
Authenticate the Supabase CLI, link the production project, and run this exact command before deploying the image feature:
  npx supabase@latest db query --linked --file supabase/migrations/20260721000100_configure_listing_images_bucket.sql
Then verify listing_image_assets and public_listing_images have RLS enabled with no anon/authenticated grants, the bucket settings match the limits above, and no broad storage.objects insert/update/delete policy exists.
The migration adopts valid legacy UUID images that are both referenced by a listing and present in this project's bucket. Inventory any remaining unreferenced bucket objects separately before deleting them.
For an upgrade with existing listing images, pause listing writes during the migration/deployment cutover and rerun the idempotent migration immediately after the new deployment before restoring writes. This prevents the previous app version from creating an untracked image row between migration and deployment.
Vercel Preview also runs with NODE_ENV=production. Give Preview a separate Supabase project and credentials if upload testing is required there; otherwise leave storage credentials unset so Preview uploads stay disabled.
Never expose the secret/service-role key with a NEXT_PUBLIC_ variable or commit it to source control.
Never run db:seed, db:seed:listings, or db:reset against production after launch; those commands are deliberately restricted to local PostgreSQL.

Important notes

- Do not keep production secrets in the developer's accounts.
- Do not reuse the developer's local database for production.
- Do not deploy production from the old GitHub repository after the move.
- The owner should own the GitHub repository, Vercel project, Supabase project, Ably app, domain, DNS, and email provider.
- The developer can stay as a collaborator only if the owner wants ongoing maintenance support.
